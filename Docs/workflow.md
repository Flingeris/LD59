# Workflow

## Goal
Codex должен помогать разрабатывать Bellgrave маленькими, проверяемыми шагами.

Он не должен:
- сразу генерировать большие куски проекта;
- ломать текущий архитектурный каркас;
- добавлять лишние системы без запроса;
- уходить в сложные абстракции раньше времени.

Цель процесса:
- быстро собирать MVP;
- сохранять читаемость кода;
- оставлять проект понятным для автора;
- не терять контроль над игровым циклом и архитектурой.

---

## Core Rule
Любая новая задача делается в порядке:

1. понять задачу
2. предложить план
3. дождаться подтверждения или правок
4. реализовать только согласованный объём
5. кратко перечислить, что изменено
6. при необходимости предложить следующий маленький шаг

Codex не должен перепрыгивать через этот порядок.

---

## Scope Discipline
Проект джемовый.

Главный приоритет:
- скорость;
- понятность;
- рабочий результат;
- простое расширение после джема.

Codex должен предпочитать:
- маленькие решения;
- минимально достаточную архитектуру;
- code that is easy to inspect manually.

Codex не должен:
- вводить сложные паттерны без явной пользы;
- дробить код на избыточное количество слоёв;
- делать “production enterprise architecture” для MVP.

---

## Task Format
Каждая задача должна быть маленькой и конкретной.

Хороший формат задачи:
- одна система;
- одна фаза;
- один экран;
- один поток действия;
- один рефакторинг-кусок.

Плохой формат:
- “сделай всю ночную фазу”
- “сделай всю экономику”
- “сделай всю боёвку”
- “сделай весь UI”

Если задача слишком широкая, Codex должен сначала разбить её на 3–7 маленьких шагов.

---

## Required Response Format For Feature Tasks
Когда пользователь даёт задачу на реализацию, Codex должен отвечать так:

### 1. Short understanding
Кратко сформулировать, что нужно сделать.

### 2. Implementation plan
Дать пошаговый план без кода.
Обычно 3–7 шагов.

### 3. Risks / notes
Если есть архитектурный риск, назвать его коротко.

### 4. Wait for approval
Не писать реализацию раньше, чем план согласован.

---

## Required Response Format After Approval
После согласования плана Codex должен:

1. реализовать только согласованный шаг или объём;
2. не добавлять новые фичи по собственной инициативе;
3. не менять архитектуру шире задачи;
4. в конце коротко перечислить:
   - какие классы добавлены;
   - какие классы изменены;
   - что теперь работает;
   - что ещё пока не сделано;
   - какой следующий логичный маленький шаг.

---

## Architecture Respect
Codex обязан учитывать текущий каркас проекта.

В проекте уже есть:
- `Main` как центральный gameplay orchestrator;
- `ServiceMain` как bootstrap;
- `G` как лёгкий глобальный реестр;
- `CMS` как источник defs;
- `AudioSystem` как отдельный сервис;
- `HUD` и `UI` как presentation layer.

Поэтому Codex не должен:
- ломать этот каркас без явного запроса;
- заменять его на DI-framework;
- переносить всю логику в UI;
- хардкодить контент в `Main`.

---

## Main Protection Rule
`Main` должен оставаться оркестратором.

Codex не должен без необходимости помещать в `Main`:
- боевые вычисления;
- контентные таблицы;
- UI-детали;
- расчёт цен и баланса;
- сложную внутреннюю логику отдельных подсистем.

Если новая логика оформляется как отдельная понятная система — её нужно выносить.

---

## UI Rule
UI только отправляет intent и отображает состояние.

UI не должен:
- сам списывать ресурсы;
- сам решать, можно ли звонить в колокол;
- сам спавнить юнитов;
- сам завершать фазу.

Правильный поток:
UI -> Main / target system -> validation -> state change -> UI refresh

---

## Content Rule
Контент должен жить в defs / CMS, а не в логике рантайма.

Если задача связана с:
- колоколами;
- юнитами;
- врагами;
- волнами;
- апгрейдами,

Codex должен по умолчанию склоняться к `Def`-подходу.

---

## Presentation Simplicity Rule
Для простых visual-задач Codex должен предпочитать простые и явные решения.

Предпочтительно:
- маленькие MonoBehaviour с одной обязанностью;
- prefab-driven setup;
- явное scene размещение;
- код, который только обновляет visual state.

Нежелательно:
- строить “универсальную” presentation-систему без реальной необходимости;
- собирать длинные chains из presenter/view/driver/helper для простой полоски, иконки или world indicator;
- решать layout магией, если его можно явно задать в prefab.

Если задачу можно надёжно решить двумя маленькими понятными компонентами, это лучше, чем один “умный” комбайн.

---

## Step Size Rule
Предпочтительный размер шага:
- 1 новая система;
- или 1 законченный поток;
- или 1 рефакторинг-кусок.

Примеры хороших шагов:
- сделать `RunState`
- сделать переключение day/night
- сделать `TryRingBell`
- сделать базовый `ResourceSystem`
- сделать один lane combat flow
- сделать экран конца ночи
- упростить один presentation flow без затрагивания остального gameplay

---

## Refactoring Rule
Рефакторинг — это отдельный вид задачи.

Codex не должен под видом “реализации фичи”:
- массово переименовывать код;
- переносить всё по папкам;
- менять половину проекта без запроса;
- переписывать несколько систем сразу, если проблема локальная.

Если нужен рефакторинг, он должен быть отдельной итерацией:
1. перечислить проблемы
2. предложить план чистки
3. после подтверждения — внести изменения

---

## Debugging Rule
При ошибках Codex должен:
1. сначала локализовать проблему;
2. назвать вероятную причину;
3. предложить минимальный фикс;
4. не перестраивать систему целиком без необходимости.

При отладке предпочтительны:
- точечные проверки;
- явные guard checks;
- краткие debug logs в ключевых местах;
- минимальные изменения.

---

## Communication Rule
Codex должен отвечать прагматично и коротко.
Без лишней теории, если она не нужна для решения.

При этом он должен:
- явно обозначать, что уже сделано;
- что осталось;
- где границы текущего шага.

---

## Generated Files Rule
Codex не должен без явной необходимости читать, анализировать, изменять или включать в отчёт автогенерируемые файлы проекта.

В первую очередь это относится к:
- `Assembly-CSharp.csproj`
- другим `.csproj`
- `.sln`
- автоматически пересобранным IDE/project-файлам
- прочим generated-файлам, которые не являются смысловой частью gameplay-задачи

Правила:
- если задача не связана напрямую со сборкой, конфигурацией проекта, пакетами, asmdef или IDE — такие файлы нужно игнорировать;
- нельзя тратить контекст на анализ автогенерируемых project-файлов без причины;
- нельзя предлагать изменения в `csproj`, если задача относится к gameplay, UI, фазам, боёвке, прогрессии или контенту;
- в отчёте по задаче не нужно перечислять generated-файлы как значимые изменения, если они поменялись автоматически;
- если Unity или IDE сами перегенерировали такие файлы, это побочный шум, а не смысловая часть реализации.

Исключения:
- задача явно про build configuration;
- задача явно про asmdef / package setup / analyzer / nullable / compiler settings;
- задача явно требует правки project-level configuration.

---

## Bellgrave-Specific Working Mode
Для этой игры Codex должен помнить:

### Day phase
Небольшая мета-фаза:
- начисление ресурсов;
- покупка улучшений;
- подготовка ночи.

### Night phase
Основной gameplay:
- звон в колокола;
- расход faith;
- вызов защитников;
- удержание кладбища;
- волна врагов.

### Defeat pressure
Главная точка давления — состояние кладбища, а не HP игрока.

### Main player decision
Главное решение игрока ночью:
- когда звонить;
- в какой колокол звонить;
- тратить ли веру сейчас;
- куда переместить хранителя;
- что временно игнорировать.

Codex не должен терять этот игровой фокус при реализации новых задач.

---

## Preferred Prompting Pattern
Для каждой новой задачи рекомендуется такой паттерн:

1. контекст
2. конкретная цель
3. ограничения
4. просьба дать план
5. после согласования — просьба реализовать

Пример структуры:

Context:
- проект Unity
- текущая архитектура: Main / ServiceMain / G / CMS
- игра Bellgrave
- текущая стадия: есть конкретный working slice

Task:
- сделать один конкретный шаг

Constraints:
- UI не должен сам спавнить юнитов
- контент через defs
- Main остаётся оркестратором
- без оверинжиниринга
- не трогать unrelated systems

Request:
- сначала дай план реализации на 3–7 шагов без кода

---

## What Codex Should Optimize For
При принятии решений Codex должен оптимизировать под:
1. clarity
2. speed of implementation
3. ease of manual inspection
4. ease of later expansion
5. respect for existing architecture

Не оптимизировать под:
- абстрактную “архитектурную красоту”;
- преждевременную универсальность;
- сложность ради гибкости, которая пока не нужна.

---

## Anti-Patterns
Плохой результат, если Codex:
- делает жирный `Main`;
- пишет gameplay прямо в кнопках UI;
- размазывает state по сцене;
- хардкодит баланс в случайных классах;
- пишет сразу огромную фичу;
- делает большой рефакторинг без отдельного согласования;
- добавляет новые сущности, которые автор потом не сможет легко читать;
- тратит контекст на generated-файлы вместо реальных runtime/source-файлов задачи;
- лезет в `csproj` и другие project-файлы без прямой необходимости;
- для простой visual-задачи строит тяжёлую “универсальную” систему.

---

## End Of Each Iteration
В конце каждой итерации Codex должен коротко писать:

- что реализовано;
- какие файлы изменены;
- что теперь работает;
- что является следующим логичным шагом.

Без длинных отчётов и без лишнего повторения кода.

---

## Bellgrave Night-Loop Priority
Для текущей итерации Bellgrave Codex должен в первую очередь усиливать новый night-management loop.

Приоритетный порядок:
1. keeper movement
2. points of interest
3. action gating by keeper position
4. faith collection during night
5. persistent breakthrough enemies
6. timer / wave readability
7. progression updates

Нельзя по умолчанию откатываться к старому паттерну, где bell gameplay живёт как изолированная HUD-кнопка без пространственного ограничения.
